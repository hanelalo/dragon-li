# Design: change-00-mvp-consistency-and-streaming-fixes

## 1) 聊天真流式改造

### 当前问题

- 传输层先读取完整响应体，再解析并批量 emit，导致 UI 感知为“伪流式”。

### 方案

- 在 `chat_provider` 增加流式读取能力，按 SSE 行增量解析 `data:` 事件。
- `chat_send` 在每次解析到 `delta` 时立即通过 `chat_stream_event` 发射。
- 收到终止事件后发 `done`，并返回最终聚合文本。

### 成功判定

- 首个 `delta` 事件在响应未结束前即可抵达前端。
- 前端能看到逐段增长，而不是末尾一次性刷出。

## 2) 配置变更检测口径统一

### 当前问题

- 归档设计写“文件监听”，实现是“调用时检测外部变化”。

### 方案

- 文档统一为“显式检查外部变更（check/apply）”。
- 暂不引入文件监听器，避免打断当前 P0 主线。

### 成功判定

- `openspec` 与 `docs` 不再出现“已实现文件监听”表述。

## 3) 软删除策略文案统一

### 当前问题

- “删除默认自动软删除”存在语义歧义。

### 方案

- 统一改为：“用户触发删除操作时，默认执行软删除（写入 `deleted_at`）”。
- 覆盖 `docs/05`、`docs/06`、`docs/10` 及相关变更文档引用。

### 成功判定

- 所有关键文档中的软删除策略语义一致且无“自动触发删除”歧义。

## 4) `DB_BUSY` 产出与重试

### 当前问题

- 文档定义了 `DB_BUSY`，代码缺少稳定错误映射与可验证重试路径。

### 方案

- 在 `sqlite_store` 错误映射识别 SQLite busy/locked，返回 `DB_BUSY`。
- 在数据库写入关键路径（至少 request log 与核心会话写入）增加有限重试：
  - 最大 2 次
  - 退避 `500ms -> 1500ms`
- 非 busy 错误保持原行为。

### 成功判定

- busy 场景返回 `DB_BUSY`。
- 重试行为与文档定义一致。

## 风险与缓解

- 风险：流式改造引入行为变化，影响现有调用方。
- 缓解：保持 `chat_send` 输出结构兼容，仅调整事件发送时机；补回归测试。
